# CV review — strengths & weaknesses

Use after [cv-analyze.md](cv-analyze.md) when the user wants feedback, not only rewrites. Keep tone **direct and actionable**.

## Objective

Separate **what to keep** from **what to fix**, so rewrites preserve credibility and only change what matters.

## Positive points (what to reinforce)

Look for:

- **Concrete outcomes:** metrics, scale, before/after, user/system scope
- **Ownership language:** led, designed, owned, shipped — not only “participated”
- **Technical depth:** decisions, trade-offs, constraints — not only tool names
- **Seniority signals:** architecture, cross-team scope, production/incident ownership
- **Clean structure:** scannable bullets, consistent tense, reasonable length
- **Credible alignment:** skills and experience match the headline and target role

Document strengths briefly so later edits **do not** water down what already works.

## Negative points (what to fix)

Group by severity when useful:

| Severity | Examples |
|----------|----------|
| **High** | No impact in any bullet; summary is generic; wrong chronology; critical info missing (dates, role title) |
| **Medium** | Weak verbs; missing metrics where plausible; skills disconnected from experience |
| **Low** | Wording polish; minor redundancy; one weak bullet in an otherwise strong role |

Common issues:

- Vague bullets (“responsible for”, “worked on”, “helped with”) without outcome
- Feature lists with no business or user effect
- Buzzwords without proof
- Too much junior history or education relative to seniority
- Missing links (GitHub, portfolio) when claiming public work

## Output format

**Strengths**

- Bullet list of 3–7 items (what is already strong)

**Weaknesses**

- Bullet list grouped by section or severity
- Each item: **issue** + **why it hurts** (one line)

**Priority actions**

- Top 3–5 fixes (ordered by impact)
