# AI / ML engineer — positioning examples

Use when the user’s primary track is **applied ML**, **MLOps**, **LLM/RAG products**, or **AI engineering** (shipping models and pipelines in production). Override every pattern with their problem domain and proof of production ownership.

## What strong AI / ML positioning looks like

- Clarifies **shipping ML in production** vs. only research: pipelines, evaluation, latency, monitoring, safety constraints.
- Names **problem types** (ranking, NLP, CV, LLM apps, retrieval) when accurate.

## Weak → strong (table)

| Weak | Strong |
|------|--------|
| AI enthusiast, ChatGPT user | ML engineer: **production model deployment and monitoring** — owned feature pipelines and A/B evaluation for recommendation and ranking models serving [scale] |
| Machine learning engineer | Applied ML engineer focusing on **LLM-powered products** — RAG architecture, evaluation harnesses, and latency-aware serving — not prompt tips only |

## Content pillar ideas

- Evaluation beyond accuracy: offline vs. online metrics, drift, bias awareness
- MLOps: reproducibility, versioning, deployment patterns
- LLM apps: retrieval quality, guardrails, cost/latency trade-offs (when experience exists)
- Collaboration with backend/product on **real product constraints**

## Headline patterns

- `ML Engineer \| [PyTorch/TF…] \| [NLP / CV / recsys / LLM apps] \| production & evaluation`
- `Applied AI Engineer \| [domain] \| shipping [LLM/RAG/agents] with measurable outcomes`
