# Insight extractor

Turn raw experience into **valuable, transferable** insights — not play-by-play.

## Objective

Answer: *What would a peer learn from this that they could not get from documentation?*

## Input types

- Technical decisions
- Bugs / failures
- Architecture choices
- Performance issues
- Process / team dynamics (when grounded in real events)

## Process

1. **What happened?** — one sentence, concrete
2. **Why did it happen?** — root cause, not blame
3. **What did you learn?** — transferable lesson
4. **What would you do differently?** — optional; shows maturity

## Output

- One **clear, transferable** insight
- **Not** obvious ("communication is important" without a specific mechanism)
- Tied to **your** context so it stays credible

## Quality bar

- If a junior could post the same insight without having shipped anything, it is too generic — add specificity (scale, constraint, failure mode).

Use the insight as the spine for **Insight** + **Expansion** in [post-creation-engine.md](post-creation-engine.md).
