# Backend / API — positioning examples

Use when the user’s primary track is **backend**, **APIs**, **services**, or **distributed systems**. Override every pattern with their real languages, data stores, and scale signals.

## What strong backend positioning looks like

- Centers **reliability, data, or scale** (latency, consistency, throughput, correctness), not “I know Java/Go/Python.”
- Mentions **boundaries you owned**: APIs, services, data stores, migrations, event-driven flows.

## Weak → strong (table)

| Weak | Strong |
|------|--------|
| Backend developer with 5 years in Node and PostgreSQL | Backend engineer specializing in **high-throughput APIs and PostgreSQL at scale** — led partitioning and read-path optimization for 10M+ daily requests |
| Experienced in microservices | Senior backend engineer: **event-driven architecture and service boundaries** — designed async pipelines replacing batch jobs, cutting processing time from hours to minutes |

## Content pillar ideas

- API design: versioning, idempotency, error contracts
- Data modeling, migrations, consistency vs. availability trade-offs
- Observability: what you actually debug in production (traces, metrics, logs)
- Performance and cost: profiling, query plans, caching layers

## Headline patterns

- `[Seniority] Backend Engineer \| [Go/Java/Node…] \| APIs · [Postgres/Redis/Kafka…] · [domain: e.g. payments · marketplaces]`
- `Staff Engineer \| Distributed systems \| [ownership area: e.g. billing platform · search]`
