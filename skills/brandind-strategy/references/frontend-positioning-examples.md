# Frontend / client — positioning examples

Use when the user’s primary track is **frontend**, **client**, or **UI engineering**. Override every pattern with their real stack, domains, and outcomes.

## What strong frontend positioning looks like

- Names a **product or UX problem domain** (e.g. real-time dashboards, design systems at scale, complex forms, AI-assisted UIs), not only a framework list.
- Separates **architecture and trade-offs** from tutorial-level skills.

## Weak → strong (table)

| Weak | Strong |
|------|--------|
| Experienced React developer with strong CSS skills | Senior Frontend Engineer focused on **design systems and performance** for high-traffic B2B apps — shipped unified component libraries used by 15+ product teams |
| Passionate about modern JavaScript | Staff-level frontend engineer: **monorepos, module boundaries, and release safety** for teams shipping weekly |

## Worked positioning statements

### Senior Frontend Engineer (React-focused)

**Weak:**
> Experienced frontend developer with skills in React, TypeScript, and modern web technologies.

**Strong:**
> Senior Frontend Engineer specializing in complex, real-time product interfaces — particularly at the intersection of AI tooling and frontend architecture. 8 years shipping production systems across creator economy, PropTech, and HR tech.

### Frontend Engineer → Staff/Principal

**Weak:**
> Staff-level engineer with broad experience across the stack.

**Strong:**
> Staff Frontend Engineer who makes frontend architecture boring — by establishing patterns and systems that let 10+ engineers ship without coordination overhead. Focused on design systems, monorepo health, and performance at org scale.

### Frontend Engineer → Tech Lead

**Weak:**
> Senior engineer transitioning into leadership.

**Strong:**
> Senior Frontend Engineer moving into tech lead roles. I write less code than I used to — and the code my team writes is better because of it. Focused on architecture decisions, code review culture, and performance ownership.

### Frontend + AI product UIs

**Strong:**
> Senior Frontend Engineer who builds the UI layer for AI products. Specializes in real-time interfaces, streaming state, and the UX of uncertainty — the specific problems that appear when your backend is non-deterministic.

## Content pillar ideas

- UI architecture (state, boundaries, when to split apps)
- Performance that users feel (not only Lighthouse)
- Collaboration with design/product; accessibility and real constraints
- If relevant: **AI product UIs** (streaming, uncertain state, permissions)

### Example pillar set (senior frontend)

**Pillar 1: Frontend Architecture** — Monorepos, design systems, component boundaries, state architecture: when and why X over Y.

**Pillar 2: AI-Integrated UIs** — Streaming, uncertain state, RBAC in multi-tenant AI apps.

**Pillar 3: Performance in real products** — Bundle size, hydration, caching, virtualization — real user impact.

**Pillar 4: Senior engineer mindset** (optional) — Ownership, trade-offs, code review, complexity — observations, not generic advice.

## Headline patterns

- `[Seniority] Frontend Engineer \| [2–3 stack signals] \| [domain: e.g. design systems · real-time · fintech UIs]`
- `Staff Frontend \| [scale signal] \| [React/TS or stack]`

## Opinion examples (frontend niche)

- "Most frontend performance problems are architecture problems in disguise."
- "Design systems fail because they're built for designers, not for the engineers who maintain them."
- "The hard part of real-time UIs isn't the real-time layer — it's managing expectations when state is uncertain."
